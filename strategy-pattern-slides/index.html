Strategy Design Pattern

Interactive deck with JavaScript and Java examples

Behavioral design pattern

Strategy moves changing algorithms out of business workflow.

The Strategy pattern lets you define a family of algorithms, put each one in its own class or object, and make them interchangeable at runtime without changing the client workflow.

ContextCheckout, shipping, pricing, sorting, auth, tax.
Strategy contractA stable method such as calculate() or pay().
Concrete strategiesVIP discount, regular discount, card payment, PayPal payment.
Runtime choicePick behavior from config, user type, feature flag, or request.

Main idea

Replace conditional algorithm selection with polymorphic behavior.

Design goal

Keep business workflow stable while algorithms vary independently.

Interview signal

You know when to use composition instead of growing if/else blocks.

Definition

Strategy is composition around a stable behavior contract.

Participants

  • Context: object that uses a strategy.
  • Strategy: interface or expected method shape.
  • Concrete strategy: one implementation of the algorithm.
  • Client: chooses which strategy to inject.

Without Strategy

A context has conditionals for every algorithm: if vip, if new, if coupon. Each new rule modifies tested workflow code.

With Strategy

The context calls strategy.execute(). New rules are added as new strategy classes or objects.

SOLID link

Strategy supports Open Closed Principle and Dependency Inversion.

Pattern family

Behavioral pattern. It changes what an object does by composing behavior.

Behavior varies Workflow stays stable Runtime selection Composition over inheritance

Example

Before and after

Move the changing rule behind a strategy.

Here the checkout workflow should not change every time the business adds a new discount policy.

Before: conditional business rules

class CheckoutService {
  checkout(cart, customer) {
    let discount = 0;

    if (customer.type === "VIP") {
      discount = cart.total() * 0.20;
    } else if (customer.type === "NEW") {
      discount = cart.total() * 0.10;
    } else if (customer.hasCoupon) {
      discount = cart.total() * 0.05;
    }

    return cart.total() - discount;
  }
}

After: interchangeable strategy

class CheckoutService {
  constructor(discountStrategy) {
    this.discountStrategy = discountStrategy;
  }

  checkout(cart, customer) {
    const discount = this.discountStrategy.calculate(cart, customer);
    return cart.total() - discount;
  }
}

class VipDiscount {
  calculate(cart) {
    return cart.total() * 0.20;
  }
}

Change isolated

Add FestivalDiscount without editing checkout.

Test focused

Test each strategy separately, then test checkout with a fake strategy.

Business readable

Policies become named objects the product team can discuss.

Usage

When and where Strategy works best.

Use Strategy when

  • You have multiple ways to perform the same business action.
  • Algorithms change independently from the main workflow.
  • A class has repeated if, else if, or switch branches for behavior.
  • You need to choose behavior at runtime.
  • You want to test each rule without booting the full application.

Use Strategy in

  • Discount, tax, commission, and pricing rules.
  • Payment providers and refund logic.
  • Shipping cost calculators.
  • Authentication and authorization policies.
  • Sorting, ranking, recommendation, and validation algorithms.

Do not force it

If there is only one simple algorithm and no sign of variation, a normal function is clearer. Strategy is useful when behavior has meaningful alternatives or future change is likely.

Tradeoffs

Pros, cons, and business efficiency.

Pros

  • Removes large conditional blocks.
  • Improves unit testing.
  • Adds new behavior without editing stable workflow.
  • Makes business policies explicit.

Cons

  • More classes or objects.
  • More wiring at runtime.
  • Can overcomplicate simple cases.
  • Bad strategy names can make code harder to navigate.

Business efficiency

  • Campaign rules can ship faster.
  • Provider changes are less risky.
  • Teams can own separate policy classes.
  • Regression tests become smaller and cheaper.
