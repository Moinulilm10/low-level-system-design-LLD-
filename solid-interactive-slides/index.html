Low level design

SOLID makes business code easier to change.

Think of SOLID as five pressure valves for object-oriented design. Each principle reduces a different kind of change pain: mixed responsibilities, rigid inheritance, fragile abstractions, oversized interfaces, and hard-coded dependencies.

Requirement changesDiscounts, payment, notifications, reporting.
Design pressureClasses grow and dependencies multiply.
SOLID choiceSeparate, extend, substitute, split, invert.
Business resultLower regression risk and faster releases.
Interview signalYou can trade off design cost versus change cost.

When to use

Use SOLID when business rules change often, when code has many collaborators, or when testing setup feels painful.

Where to use

Best fit: services, domain models, use cases, integrations, repositories, policies, validators, and workflow orchestration.

When to pause

Avoid over-design for tiny scripts, throwaway prototypes, and stable data holders with no meaningful behavior.

The acronym

Five principles, five design questions.

S

Single Responsibility

Does this class have only one reason to change?

O

Open Closed

Can new behavior be added without editing stable code?

L

Liskov Substitution

Can every subtype safely replace its parent type?

I

Interface Segregation

Do clients depend only on methods they actually use?

D

Dependency Inversion

Does business logic depend on abstractions instead of concrete tools?

Important interview framing

SOLID is not a rule book for making more classes. It is a way to decide where change should be isolated. Strong answers mention the expected change, the cost of abstraction, and how the design improves testability or release safety.

Example

Visualization

One checkout flow before and after SOLID.

A common business case: create an order, calculate discount, charge payment, send receipt, and persist the result.

Before: one service knows everything

class CheckoutService {
  checkout(cart, customer) {
    const total = cart.items.reduce(
      (sum, item) => sum + item.price,
      0
    );

    let discount = 0;
    if (customer.type === "VIP") discount = total * 0.15;
    if (customer.type === "NEW") discount = total * 0.05;

    const payable = total - discount;

    Stripe.charge(customer.card, payable);
    Database.save({ cart, customer, payable });
    Email.send(customer.email, "Receipt");

    return payable;
  }
}

After: business policy is separated

class CheckoutService {
  constructor(
    discountPolicy,
    paymentGateway,
    orderRepository,
    notifier
  ) {
    this.discountPolicy = discountPolicy;
    this.paymentGateway = paymentGateway;
    this.orderRepository = orderRepository;
    this.notifier = notifier;
  }

  checkout(cart, customer) {
    const total = cart.total();
    const discount = this.discountPolicy.discountFor(cart, customer);
    const payable = total - discount;

    this.paymentGateway.charge(
      customer.paymentMethod,
      payable
    );
    this.orderRepository.save({ cart, customer, payable });
    this.notifier.receipt(customer, payable);

    return payable;
  }
}

Change isolated

A new discount rule changes a policy object, not payment or persistence code.

Testing gets smaller

Checkout tests can use fake dependencies and focus on orchestration.

Business rules become visible

Policies, gateways, repositories, and notifiers map to language the business understands.

Application

When and where SOLID pays for itself.

Use it when

  • Rules are expected to change: pricing, tax, eligibility, commission, fraud checks.
  • External tools can change: payment provider, email vendor, database, cache, queue.
  • You need unit tests around business behavior without real infrastructure.
  • Several teams touch the same module and regressions are expensive.
  • A class is becoming a "god object" with unrelated reasons to change.

Use it in

  • Domain services: checkout, billing, booking, inventory, loan approval.
  • Policy objects: discount, validation, authorization, scoring.
  • Application services: workflow coordination across multiple dependencies.
  • Adapters: database repositories, API clients, payment gateways.
  • Plugin-style features: reports, exporters, importers, notification channels.

Practical rule

Start simple. Introduce abstractions at the boundary of repeated change. If the code changes once, a direct implementation is fine. If it changes for the third time in the same direction, the design is asking for a SOLID move.

Tradeoffs

Pros and cons you should say out loud.

Pros

  • Lower regression risk because changes are localized.
  • Better testability because dependencies can be mocked or faked.
  • Cleaner business language because rules become named objects.
  • Easier extension because new strategies, adapters, and policies can be added.
  • More maintainable modules because responsibilities are smaller.

Cons

  • More classes, files, and indirection can slow beginners down.
  • Premature abstraction can hide simple logic behind ceremony.
  • Too many tiny interfaces can make navigation harder.
  • Bad abstractions become costly because many classes depend on them.
  • Runtime wiring or dependency injection can feel heavier in small apps.

Can you name the business variation? If yes, the abstraction may be worth it.

Will this code change independently from its neighbors? Separate it.

3
Mock test

Is unit testing blocked by a database, network, clock, or file system? Invert that dependency.

4
Readability test

If SOLID makes the simple path harder to read, wait until the design pressure is real.

Business efficiency

SOLID turns change into smaller work packages.

How it helps business logic

  • Rules can be reviewed in isolation by engineers and domain experts.
  • Feature flags and A/B pricing policies are easier to add as strategies.
  • Vendor migrations are safer because adapters contain infrastructure details.
  • Regression testing is cheaper because small units have clear contracts.
  • Teams can work in parallel when modules have stable boundaries.

Business value sentence for interviews

"I use SOLID to protect volatile business rules from infrastructure and unrelated workflow changes, so the team can ship policy updates quickly without retesting every external integration."

Interview practice

Questions, situations, and answer cues.

Concept questions
  • Explain each SOLID principle with one business example.
  • What is the difference between SRP and separation of concerns?
  • How does OCP relate to the strategy pattern?
  • Why is LSP more than just inheritance syntax?
  • How do DIP and dependency injection differ?
Situation questions
  • A checkout class calculates tax, charges card, saves orders, and sends email. Refactor it.
  • A report service has many if type === ... blocks. Add CSV without editing old logic.
  • A ReadOnlyFile inherits from File but throws on write(). Identify the problem.
  • A printer interface forces scanner-only devices to implement print(). Fix the design.
  • An order service creates new StripeClient() inside methods. Make it testable.

Strong answer pattern

Describe the smell, name the principle, identify the volatile dependency or rule, show a smaller abstraction, and mention the tradeoff. Example: "This violates DIP because checkout owns Stripe creation. I would depend on a PaymentGateway abstraction and inject a Stripe implementation. That makes tests faster and lets us add PayPal later, but I would avoid this abstraction if we know the product will only ever use one provider."

Revision notes

Quick memory hooks.

One-line definitions

  • SRP: one reason to change.
  • OCP: open for extension, closed for modification.
  • LSP: subtype must honor parent behavior.
  • ISP: small client-specific interfaces.
  • DIP: high-level policy depends on abstractions.

Smell to principle map

  • God class: SRP.
  • Repeated type checks: OCP.
  • Subclass throws unsupported errors: LSP.
  • Fat interface with empty methods: ISP.
  • Business service creates infrastructure clients: DIP.
