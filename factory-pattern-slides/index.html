Factory Design Pattern

Interactive deck with JavaScript and Java examples

Creational design pattern

Factory centralizes object creation so business code stays clean.

The Factory pattern hides creation logic behind a method, class, or interface. The caller asks for an object by intent, and the factory decides which concrete class to create.

Client need"Create a notification sender", "Create a payment gateway".
FactoryChooses the right class and construction details.
ProductConcrete object such as EmailSender or StripeGateway.
Business serviceUses the product through a stable contract.

Main idea

Move object creation decisions out of business workflow.

Design goal

Business services depend on what they need, not how it is constructed.

Interview signal

You can separate creation complexity from domain behavior.

Factory family

Factory has three common forms.

Simple Factory

A method or class creates products based on input. It is practical, common, and easy to understand, but not always considered a formal GoF pattern.

Factory Method

A base class defines a creation method, and subclasses decide which product to instantiate.

Abstract Factory

A factory creates families of related products, such as UI controls for web and mobile themes.

Practical rule

Use Simple Factory for straightforward selection, Factory Method when subclasses should control creation, and Abstract Factory when products must be created as a compatible family.

Example

Before and after

Replace scattered constructors with one creation boundary.

Here an order service should not know how every notification channel is constructed.

Before: creation mixed with workflow

class OrderService {
  confirm(order, channel) {
    let sender;

    if (channel === "email") {
      sender = new EmailSender(process.env.SMTP_KEY);
    } else if (channel === "sms") {
      sender = new SmsSender(process.env.SMS_TOKEN);
    } else {
      sender = new PushSender(process.env.PUSH_KEY);
    }

    sender.send(order.customer, "Order confirmed");
  }
}

After: factory owns creation

class NotificationFactory {
  static create(channel) {
    const creators = {
      email: () => new EmailSender(Config.smtpKey),
      sms: () => new SmsSender(Config.smsToken),
      push: () => new PushSender(Config.pushKey)
    };

    return (creators[channel] || creators.email)();
  }
}

class OrderService {
  confirm(order, channel) {
    const sender = NotificationFactory.create(channel);
    sender.send(order.customer, "Order confirmed");
  }
}

Usage

When and where Factory works best.

Use Factory when

  • Object creation has conditions, configuration, or setup steps.
  • The caller should not know concrete class names.
  • You need to choose a product based on request, tenant, provider, or environment.
  • You want to keep constructors out of business services.
  • Related objects must be created consistently.

Use Factory in

  • Payment gateway selection.
  • Notification sender creation.
  • Report exporter creation.
  • Database or cache client creation.
  • UI component families and platform-specific services.

Do not force it

If construction is a single obvious new with no branching, setup, or future variation, direct construction is clearer.

Tradeoffs

Pros, cons, and business efficiency.

Pros

  • Centralizes creation logic.
  • Reduces coupling to concrete classes.
  • Improves test setup and replacement.
  • Makes provider selection explicit.

Cons

  • Adds extra classes or methods.
  • Can hide too much if factory names are vague.
  • Simple factories can grow into large switch statements.
  • Incorrect factory ownership can create a service locator smell.

Business efficiency

  • Provider changes happen in one place.
  • Tenant-specific setup is easier.
  • Release risk drops around integrations.
  • Services focus on business workflow.
