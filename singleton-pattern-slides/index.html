Singleton Design Pattern

Interactive deck with JavaScript and Java examples

Creational design pattern

Singleton guarantees one shared instance with global access.

The Singleton pattern restricts a class to exactly one instance and gives the application a controlled way to access it. It is useful for shared infrastructure, but risky when it becomes hidden global state.

Private creationExternal code cannot freely create instances.
Static holderThe class stores its one instance.
Access methodCall getInstance() or import a module instance.
Shared objectLogger, config, metrics, cache, connection manager.

Main idea

One object coordinates shared state or shared infrastructure.

Design goal

Prevent duplicate expensive or inconsistent instances.

Required context

Concepts behind Singleton.

Static members

A static field or method belongs to the class itself, not to an individual object. Singleton usually stores the instance in a static field.

Private constructor

In Java, a private constructor prevents external code from calling new. JavaScript can simulate this with module scope or private tokens.

Interface overview

An interface is a contract. In Java, you can depend on Logger while the singleton implementation is AppLogger. JavaScript uses duck typing for the same idea.

Lazy initialization

Create the instance only when first needed. Saves startup cost but needs thread safety in multi-threaded languages.

Eager initialization

Create the instance when the class loads. Simpler and thread-safe in Java, but may create an object even if unused.

JavaScript module cache

ES modules are evaluated once per runtime and cached, so exporting one object is often the cleanest JavaScript singleton.

Example

Before and after

Replace duplicate infrastructure objects with one controlled instance.

Here multiple services repeatedly create configuration readers, causing duplicated loading and inconsistent values during runtime.

Before: repeated configuration loading

class OrderService {
  constructor() {
    this.config = new ConfigManager();
  }
}

class PaymentService {
  constructor() {
    this.config = new ConfigManager();
  }
}

class ConfigManager {
  constructor() {
    this.settings = loadEnvAndJsonFiles();
  }
}

After: shared singleton configuration

class ConfigManager {
  static instance;

  static getInstance() {
    if (!ConfigManager.instance) {
      ConfigManager.instance = new ConfigManager();
    }
    return ConfigManager.instance;
  }

  constructor() {
    if (ConfigManager.instance) return ConfigManager.instance;
    this.settings = loadEnvAndJsonFiles();
  }
}

const config = ConfigManager.getInstance();

Usage

When and where Singleton works best.

Use Singleton when

  • Exactly one instance should exist in a process.
  • Object creation is expensive and repeated creation is wasteful.
  • A shared coordinator must maintain consistent state.
  • The instance has no request-specific or user-specific state.
  • Lifecycle should be controlled in one place.

Use Singleton in

  • Application configuration reader.
  • Logger or telemetry collector.
  • Metrics registry.
  • Feature flag provider.
  • Connection pool manager, not individual DB connections.

Do not force it

Do not use Singleton for request state, user sessions, shopping carts, domain entities, or anything that needs multiple independent instances.

Tradeoffs

Pros, cons, and business efficiency.

Pros

  • Ensures one controlled instance.
  • Can reduce repeated expensive setup.
  • Provides consistent shared configuration.
  • Simple global access for infrastructure concerns.

Cons

  • Can become hidden global mutable state.
  • Can make unit tests harder.
  • Can hide dependencies from constructors.
  • Thread safety and lifecycle can be tricky.

Business efficiency

  • Configuration stays consistent.
  • Logging and metrics are centralized.
  • Expensive initialization happens once.
  • Operational behavior is easier to standardize.
1

Good Singleton: stateless or process-level infrastructure such as logger, config, metrics.

2

Bad Singleton: business data that changes per user, request, transaction, or test.
